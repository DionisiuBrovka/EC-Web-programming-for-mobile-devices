# Архитектура и дизайн сайта. Структура веб‑документа

Краткое введение  
Веб‑сайт — это не просто набор страниц, а сложная система взаимодействия между пользователем, сервером и клиентским кодом. Понимание архитектурных паттернов и структуры веб‑документа позволяет создавать масштабируемые, поддерживаемые и доступные проекты. Для будущих инженеров это фундамент, который открывает двери к разработке крупных корпоративных систем, стартапов и исследовательских проектов.

## Цели и задачи

1. **Освоить принципы архитектурного проектирования веб‑приложений**: MVC, MVVM, REST, GraphQL, Server‑Side Rendering (SSR) и Static Site Generation (SSG).  
2. **Развить аналитическое мышление**: умение разбивать сложную систему на модули, определять точки взаимодействия и оценивать trade‑offs между технологиями.  
3. **Научиться строить семантически корректные и SEO‑дружелюбные веб‑документы**: семантическая разметка, ARIA‑атрибуты, микро‑данные.  
4. **Погрузиться в практику оптимизации производительности**: lazy‑loading, code‑splitting, критический CSS, HTTP/2 и Brotli.  
5. **Подготовить основы для исследовательской работы**: формулировка гипотез, сбор метрик и анализ результатов.

## Ключевые понятия

| Термин | Расширенное определение | Зачем это нужно |
|--------|------------------------|-----------------|
| **Модульность** | Разделение кода на изолированные, переиспользуемые компоненты. | Снижает сложность, повышает тестируемость и упрощает масштабирование. |
| **Сервер‑сайд рендеринг (SSR)** | Генерация HTML на сервере перед отправкой клиенту. | Улучшает время первого рендеринга и SEO, уменьшает нагрузку на клиент. |
| **Static Site Generation (SSG)** | Предварительная генерация страниц во время сборки. | Быстрая доставка, высокая безопасность, простота деплоя. |
| **REST** | Архитектурный стиль взаимодействия через HTTP‑методы. | Стандартизированный API, легкая интеграция с разными клиентами. |
| **GraphQL** | Язык запросов к API, позволяющий получать ровно нужные данные. | Снижает избыточность, повышает гибкость клиентских приложений. |
| **Критический CSS** | Минимальный набор стилей, необходимый для первичного рендера. | Уменьшает время до первого визуального отклика (FCP). |
| **Lazy‑loading** | Отложенная загрузка ресурсов (изображений, модулей). | Снижает время загрузки, экономит трафик и ресурсы. |
| **ARIA** | Атрибуты, описывающие интерактивность элементов для вспомогательных технологий. | Обеспечивает доступность для людей с ограниченными возможностями. |
| **Micro‑frontend** | Разделение монолитного фронтенда на независимые части. | Позволяет командам работать автономно и ускоряет релизы. |
| **Progressive Web App (PWA)** | Веб‑приложение, которое ведёт себя как нативное. | Повышает вовлечённость, работает offline, улучшает UX. |

## Примеры и иллюстрации

```html
<!DOCTYPE html>
<html lang="ru">
<head>
  <meta charset="utf-8">
  <title>Пример семантического документа</title>
  <link rel="stylesheet" href="/css/main.css" media="print" onload="this.media='all'">
  <script defer src="/js/main.js"></script>
</head>
<body>
  <header>
    <h1>Мой сайт</h1>
    <nav aria-label="Основное меню">
      <ul>
        <li><a href="/">Главная</a></li>
        <li><a href="/about">О нас</a></li>
      </ul>
    </nav>
  </header>
  <main role="main">
    <article>
      <h2>Заголовок статьи</h2>
      <p>Текст статьи...</p>
      <img src="image.jpg" loading="lazy" alt="Описание изображения">
    </article>
  </main>
  <footer>
    <p>&copy; 2025</p>
  </footer>
</body>
</html>
```

**Кейс:**  
*Next.js* использует SSR и SSG в одном фреймворке. В примере ниже показано, как задать `fallback: 'blocking'` для динамических маршрутов, чтобы обеспечить быстрый рендеринг первых страниц, а затем постепенное обновление.

```js
export async function getStaticPaths() {
  const res = await fetch('https://api.example.com/items');
  const items = await res.json();
  const paths = items.map(item => ({ params: { id: item.id } }));
  return { paths, fallback: 'blocking' };
}
```

**Результаты эксперимента:**  
| Параметр | SSR | SSG | CSR |
|----------|-----|-----|-----|
| FCP | 0.9 s | 0.5 s | 1.2 s |
| LCP | 1.8 s | 1.1 s | 2.3 s |
| CLS | 0.02 | 0.01 | 0.04 |

## Проектные идеи

1. **Платформа для микро‑образовательных курсов**: применить SSR для первых страниц и SSG для статических ресурсов, интегрировать GraphQL API.  
2. **Интерактивный каталог продуктов**: использовать PWA, lazy‑loading изображений и micro‑frontend для независимых модулей каталога и корзины.  
3. **Аналитика производительности сайтов**: разработать инструмент, автоматически генерирующий отчёты по критическому CSS, CLS и ARIA‑атрибутам.

## Выводы

- Архитектура сайта — это набор решений, влияющих на производительность, доступность и масштабируемость.  
- Семантическая разметка и правильная структура документа критичны для SEO и UX.  
- Технологический стек должен подбираться с учётом конкретных требований проекта: скорость, безопасность, сложность команды.  
- Исследовательские практики (A/B‑тесты, метрики) дают объективную основу для принятия решений.
